Java Server Pages

Making Java like HTML

JSP Overview

  • A JSP page is a web page that contains Java code along with the HTML tags.
  • It does not involve the explicit writing and compilation of a Java class as it is done internally.
  • The JSP engine parses the JSP file and creates a corresponding Java file which declares a servlet class whose members map directly to elements of JSP file.
  • JSP engine then compiles the class, loads it into memory, and executes it as it would any other servlet.
  • JSP is mainly used in the presentation layer or the View and does not have any complex business logic.

Directives

  • Directives provide general information about the JSP page to the JSP engine.
  • There are three types of directives: page, include, and taglib.
  • A page directive informs the engine about the overall properties of a JSP page.
  • An include directive tells the JSP engine to include the contents of another file (HTML, JSP, etc.) in the current page.
  • A taglib directive is used for associating a prefix with a tag library.

Directives

  • A directive always starts with <%@ and ends with %>.
  • The tag names, their attributes, and their values are all case sensitive.
  • The value must be enclosed within a pair of single or double quotes.
  • A pair of single quotes is equivalent to a pair of double quotes.
  • There must be no space between the equals sign (=) and the value.

Declarations

  • Declarations declare and define variables and methods that can be used in the JSP page.
  • The variable is initialized only once when the page is first loaded by the JSP engine, and retains its value in subsequent client requests.
  • A declaration always starts with <%! and ends with %>.
  •  It can contain any number of valid Java declaration statements.
  • Since the declarations contain Java declaration statements, each variable’s declaration statement must be terminated with a semicolon.
  • <%! int count = 0; %>

Scriptlets

  • Scriptlets are Java code fragments that are embedded in the JSP page.
  • <% count++; %>
  • The scriptlet is executed each time the page is accessed
  • A scriptlet always starts with <% and ends with %>.
  • The code within the scriptlet must be valid in the Java programming language.
  • Scriplets are avoided currently in designing web applications, as they mix up HTML and Java code.

Expressions

  • Expressions act as placeholders for Java language expressions.
  • <%= count %>
  • The expression is evaluated each time the page is accessed, and its value is then embedded in the output HTML.
  • A JSP expression always starts with <%= and ends with %>.
  • Expressions must not be terminated with a semicolon.
  • Expressions can print the value of any object or any primitive data type or any arithmetic or Boolean expression or a value returned by a method call to the output stream.
  • Expressions cannot have a definition statement  or call a method whose return type is void.

Actions

  • Actions are commands given to the JSP engine.
  • They direct the engine to perform certain tasks during the execution of a page.
  • There are six standard JSP actions:
  • The jsp:include and jsp:forward enable a JSP page to reuse other web components.
  • jsp:useBean, jsp:setProperty and jsp:getProperty are related to the use of JavaBeans in JSP pages.
  • jsp:plugin instructs the JSP engine to generate appropriate HTML code for embedding client-side components, such as applets.
  • A JSP page can also have user-defined actions called custom tags.

Actions

  • The general syntax of a JSP action is:
  •     <jsp:actionName attribute-list />
  • The actionName is one of the six actions mentioned before and attribute-list represents one or more attribute-value pairs. 
  • The action names, their attributes, and their values are case sensitive.
  • The value must be enclosed within a pair of single or double quotes.
  • A pair of single quotes is equivalent to a pair of double quotes.
  • There must be no space between the equals sign (=) and the value.

Comments

  • Comments do not affect the output of a JSP page in any way but are useful for documentation purposes. 
  • The syntax of a JSP comment is:
  •    <%-- Anything you want to be commented --%>
  • A JSP comment always starts with <%-- and ends with --%>.
  • Three kinds of comments are:
  • <%-- JSP comment --%>
  • <% //Java comment %>
  • <!-- HTML comment -->

Translation Units

  • The include directive includes the contents of HTML or JSP page in other JSP page.
  • When the JSP engine generates the Java code for a JSP page, it also inserts the contents of the included pages into the servlet generating it. 
  • The set of pages that is translated into a single servlet class is called a translation unit.
  • The page directives affect the whole translation unit.
  • A variable declaration cannot occur more than once in a single translation unit.
  • The standard action <jsp:useBean> cannot declare the same bean twice in a single translation unit.

JSP Page Life Cycle

  • Creating the servlet instance: It involve the process of converting the JSP page into an instance of a servlet class.
  • Translation: The JSP engine reads a JSP page, parses it, validates the syntax of the tags used and then creates a Java file containing a public servlet class.
  • Compilation: The java file is compiled  and all the declarations, scriptlets, and expressions is validated.
  • We can force a compilation of a JSP page without execution using precompilation request parameter jsp_precompile which takes a Boolean value.
  • All of the request parameter names including the prefix jsp are reserved and must not be used for user-defined values.

JSP Page Life Cycle

  • Loading and instantiation: After successful compilation, the container loads the servlet class into memory and instantiates it.
  • Calling the JSP life-cycle methods:
  • The generated servlet class for a JSP page implements the HttpJspPage interface of javax.servlet.jsp package.
  • HttpJspPage interface extends JspPage interface which extends the Servlet interface.
  • The JspPage interface declares only two methods—jspInit() and jspDestroy().
  • HttpJspPage interface is specifically provided for JSP pages serving HTTP requests and declares only _jspService() method.

JSP Methods

  • jspInit():
  • The container calls jspInit() to initialize the servlet instance. 
  • It is called before any other method, and is called only once for a servlet instance. 
  • Usually the method does initial or one-time setup, such as acquiring resources and` in the JSP page using <%! ... %>. 
  • _jspService(): 
  • The container calls the _jspService() for each request, passing it the request and the response objects. 
  • All of the HTML elements, the JSP scriptlets, and the JSP expressions become a part of this method during the translation phase.

JSP Methods

  • jspDestroy(): 
  • When the container decides to take the instance out of service, it calls the jspDestroy() method. 
  • This is the last method that is called on the servlet instance, and it is used to clean up the resources acquired in the jspInit() method.
  • The jspInit() and jspDestroy() methods can be overridden using the JSP declaration tag <%! ... %>.
  • The _jspService() method cannot be overridden because the engine generates it automatically.

import and session attributes

  • import: The import attribute of a page directive is similar to the import statement in a Java class.
  • At the time of translation, the JSP engine inserts an import statement into the generated servlet for each of the packages declared using this attribute.
  • It can occur multiple times in a translation unit, with duplicate values being ignored.
  • session: It indicates whether or not the JSP page takes part in an HTTP session. 
  • The default value is true, in which case the JSP engine declares the implicit variable session.
  • Explicitly specifying false to the session attribute disables session in the jsp page.

errorPage and isErrorPage attributes

  • errorPage: The errorPage attribute specifies the JSP engine to delegate the exception from current JSP page to another JSP page which has the errorhandling code.
  • The errorPage value can be either a JSP page or an HTML page.
  •  isErrorPage: The isErrorPage attribute conveys whether or not the current page can act as an error handler for any other JSP page. 
  • The default value of the isErrorPage attribute is false.
  • Such page only extracts the information from the exception and generates an appropriate error message.
  •  

language and extends attributes

  • language: The language attribute specifies the language used by a page in declarations, scriptlets, and expressions. 
  • The default value is java.
  • extends: The extends attribute specifies that the supplied class be used as a base class of the generated servlet.
  • The default base class is vendor specific and is designed to work efficiently with the rest of the framework.

buffer and autoFlush attributes

  • buffer: The buffer attribute specifies the minimum size required by the output buffer that holds the generated content until it is sent to the client. 
  • The default buffer size varies from implementation, but the specification mandates it to be at least 8kb.
  • Buffer size is in kilobytes and suffix kb is mandatory.
  • The value none of the buffer sends the data directly to the client without any buffering.
  • autoFlush: The autoFlush attribute specifies whether the data in the output buffer should be sent to the client automatically as soon as the buffer is full. 
  • The default value for autoFlush is true. 
  • If it is set to false and the buffer is full, an exception is raised when we attempt to add more data to the buffer.

The isThreadSafe attribute

  • It specifies whether a page can handle multiple requests concurrently.
  • The default value of isThreadSafe attribute is true.
  • Hence the container can dispatch multiple requests to the page’s servlet concurrently in different threads.
  • If isThreadSafe is set to false, the container dispatches the requests to the page one at a time in the order they are received, equivalent of implementing the SingleThreadModel interface for the servlets.
  • If the container decides to create a pool of instances of the thread-unsafe servlet, then programmatic synchronization using synchronized block\methods with isThreadSafe set to true is better option.

The info attribute

  • The info attribute allows to specify the value of the string returned by the getServletInfo() method of the generated servlet.
  • <%@ page info="This is a sample Page. " %>
  • The default value of this attribute is implementation dependent.

contentType and pageEncoding

  • contentType: It specifies the MIME type and character encoding of the output. 
  • The default value of the MIME type is text/html; the default value of the character encoding is ISO-8859-1.
  • The MIME type and character encoding are separated by a semicolon and is equivalent to setContentType().
  • <%@ page contentType="text/html;charset=ISO-8859-1" %>
  • pageEncoding: It specifies the character encoding of the JSP page. The default value is ISO-8859-1.
  • <%@ page pageEncoding="ISO-8859-1" %>

JSP Elements Translation Rules

  • Some directives just inform the engine about the overall properties of the page.
  • Some directives are used by the JSP engine during the translation phase to generate Java code.
  • All JSP declarations become a part of the generated servlet class and are copied as it is.
  • All JSP scriptlets become a part of the generated _jspService() method and are copied as it is.
  • All JSP expressions become a part of the _jspService() method. They are wrapped inside out.print().
  • All JSP actions are replaced by calls to vendor-specific classes.
  • All JSP comments are ignored.
  • Any other text becomes part of the _jspService() method. It is wrapped inside out.write(). This text is also called template text.

Scripting Elements

  • The declarations, scriptlets, and expressions are collectively referred to as the scripting elements.
  • Order of declarations: variables and methods become members of generated servlet so order in page does not matter.
  • Order of scriptlets: Scriptlets become a part of the _jspService() method in generated servlet, variables declared in a scriptlet become local variables of the method; hence their order of appearance is important.
  • Initialization of the variables: Variables declared in JSP declarations are initialized to their default values, while the variables declared in JSP scriptlets must be initialized explicitly before they are used.

Scripting Elements

  • conditional and iterative statements: conditional and iterative statements can span across multiple scriptlets in the JSP page.
  • request-time attribute expressions: JSP expressions are not always written to the output stream of the JSP page; they can also be used to pass values to action attributes. <jsp:include page="<%=pageURL%>" />
  • escape sequences: To use a backslash and create an escape sequence in order to instruct the parser not to use them as special characters.

application and session variables

  • application: It is of type javax.servlet.ServletContext, and it refers to the environment of the web application.
  • session: The implicit variable session is declared if the value of session attribute of the page directive is true. 
  • By default, the value of the session attribute is true.
  • When the session attribute is explicitly set to false, the JSP engine does not declare the variable, and any use of the variable results in an error.

request, response and page

  • The request and response implicit variables are of type javax.servlet.http.HttpServletRequest and javax.servlet.http.HttpServletResponse, respectively.
  • They are passed in as parameters to the _jspService() method when the page’s servlet is executed upon a client request.
  • The implicit variable page is of class java.lang.Object, and it refers to the instance of the generated servlet. 
  • The variable is rarely used and is declared as:
  • Object page = this; //this refers to instance of this servlet.
  •  

pageContext implicit variable

  • pageContext is of type javax.servlet.jsp.PageContext which is an abstract class, and the JSP engine vendor provides its concrete subclass.
  • The pageContext object acts as a onestop place for managing all the other objects, both user-defined and implicit objects.
  • It provides convenience methods to get and set attributes in different scopes and for transferring requests to other resources in the web application. Methods are as follows:
  • void include(String relativeURL): Includes the output of another resource in the output of the current page. Same as ServletRequest.getRequestDispatcher().include();
  • void forward(String relativeURL): Forwards the request to another resource. Same as ServletRequest.getRequestDispatcher().forward();

out and exception implicit variables

  • out is the implicit variable of type javax.servlet.jsp.JspWriter.
  • The JspWriter class extends java.io.Writer and inherits all the overloaded write() methods. 
  • On top of it, JspWriter adds its own set of overloaded print() and println() methods for printing out all the primitive data types, Strings, and the user-defined objects in the output stream.
  • exception implicit variable is of type java.lang.Throwable. 
  • The exception variable is available to the pages that act as error handlers for other pages. 
  • The error-handler pages are those pages having page directive attribute isErrorPage set to true.

config implicit variable

  • The implicit variable config is of type javax.servlet.ServletConfig.
  • Each servlet can be passed a set of configuration parameters in the deployment descriptor, and it can retrieve the information using ServletConfig object.
  • The config variable gets the configuration parameters specific to a JSP page, declared in the deployment descriptor with a <servlet-name> associated with the element <jsp-file>. 
  • All of the initialization parameters for the named servlet will then be available to the JSP page via the page’s ServletConfig implicit object.
  • The JSP engine creates two different instances of the generated servlet class—one for accessing it as a named servlet and one for accessing it as a JSP page—and will pass each servlet instance a different ServletConfig object.
  • To use the same servlet instance an explicit mapping of the JSP page’s URL in the deployment descriptor file using the <servlet-mapping> element is carried out.

Static Inclusion

  • The contents of another file are included with the current JSP file at translation time to produce a single servlet.
  • <%@ include file="relativeURL" %>
  • <jsp:directive.include file="relativeURL" />
  • The variables, methods and implicit objects are shared across both the including as well as included pages.
  • No processing can be done at translation time, which means the file attribute value cannot be an expression.
  • The file attribute value cannot pass any parameters to the included page as requests are absent at translation time.
  • The included page may or may not be able to compile independently.
  •  

Dynamic Inclusion

  • <jsp:include>: When the JSP page is requested it sends a request to another object, and the output from that object is included in the requested JSP page.
  • <jsp:forward>: It delegates the request processing to the forwarded component. 
  • The forwarded component then sends the reply to the client
  • The standard JSP actions to implement dynamic inclusion are <jsp:include> and <jsp:forward>:
  • <jsp:include page="relativeURL" flush="true" />
  • <jsp:forward page="relativeURL" />

Include and Forward Attributes

  • The page attribute is mandatory, and must be a relative URL. It can refer to any static or dynamic web component, or can be a request-time expression.
  • The flush attribute is only valid for <jsp:include> and not for <jsp:forward>.
  • Flush attribute is optional and specifies that if the output of the current JSP page is buffered, then the buffer should be flushed before passing the output stream to the included component.
  • The default value for the flush attribute is false.

Passing parameters to dynamically included components

  • <jsp:param /> tags are used to pass parameters to dynamically included components.
  • The <jsp:include> or <jsp:forward> elements can have multiple <jsp:param> elements nested within them.
  • The included components also have access to parameters which were originally present in the request to the including component.
  • If the original parameter names are repeated in the explicitly passed parameters, then the new values take precedence over the old values.
  • The name-value pairs passed in via the <jsp:param> tag exist within the request object and are available only for the included component. 
  • After the included component has finished processing, the engine removes these values from the request object.

Sharing objects with dynamically included components

  • The dynamically included pages execute separately, so they do not share the variables and methods defined by the including page.
  • They process the same request and thus share all the objects present in the request scope.
  • The implicit variables session and application could be used to share objects with included and forwarded pages, but are not meant for sharing request-dependent values.

JavaBeans

  • A bean is a Java class object that encapsulates data in the form of instance variables. 
  • These variables are referred to as properties of the bean. 
  • The class also provides a set of methods for accessing and mutating its properties. 
  • In JSP technology, the JSP engine acts as a bean container and any class that follows the following conventions are JavaBean in JSP pages:
  • 1)  The class must have a public constructor with no arguments which allows the class to be instantiated as needed by the JSP engine.
  • 2)  For every property, the class must have two publicly accessible methods, referred to as the getter and the setter, that allow the JSP engine to access or mutate the bean’s properties.

Serialized JavaBeans

  • The JavaBeans specification also allows to persist JavaBeans in the file system as serialized objects.
  • A serialized bean is a bean instance that is converted into a data stream and stored in a file so that its attributes and values are saved permanently and can be retrieved later. 
  • The process of serialization is achieved using the standard Object Serialization mechanism of Java. 
  • The bean class should implement Serializable interface to be capable of being serialized.
  • The file that stores a serialized bean must have the extension .ser.
  • The combination of the path and the filename that stores the bean is treated as the name of the bean.
  • Serialized beans are loaded as objects using the method: java.beans.Beans.instantiate( this.getClass(). getClassLoader(),“somepackage.Someclass");

Attributes: id and scope

  • id attribute: It uniquely identifies a particular instance of a bean. It is a mandatory attribute. 
  • In the generated Java servlet, the value of id is treated as a Java language variable; hence the variable name can be used in expressions and scriptlets in a JSP page.
  • Same value for the id attribute in more than one <jsp:useBean> action within a single translation unit is not permitted.
  • scope attribute: The scope attribute specifies the scope in which the bean instance resides and is optional.
  • The existence and accessibility of JavaBeans from JSP pages are determined by the four JSP scopes: page, request, session, and application. The page scope is mostly default.

Attributes: class and type

  • class attribute: It specifies Java class of the bean instance. 
  • If the <jsp:useBean> action cannot find an existing bean in the specified scope, it creates a new instance of the bean’s class as specified by the value of the class attribute using the class’s publicly defined no-argument constructor. 
  • The class attribute must be a public non-abstract class and must have a public noargument constructor.
  • type attribute: It specifies the type of the variable declared by the id attribute.
  • The declared variable refers to the actual bean instance at request time, its type must be the same as the bean’s class, or a superclass of the bean’s class, or an interface implemented by the bean’s class.
  • The class for existing instance can be of the same name as type attribute class or its subclass.
  • The type attribute without the class or beanName attributeis useful to locate an existing bean object but doesn’t create a new instance even if an existing instance is not available.

The beanName Attribute

  • The beanName attribute specifies the name of a bean as expected by the instantiate() method of the java.beans.Beans class.
  • The beanName can refer either to a serialized bean or to the name of a class whose instance is to be created.
  • If the beanName attribute refers to a serialized bean, then bean is loaded from corresponding file that holds the bean.
  • If the beanName attribute refers to a class, the class is loaded into the memory, an instance of the class is created, and the instance is used as a bean.
  • The advantage of using beanName over class which is also used to create new instances, is that the value of the beanName attribute can also be specified as a request-time attribute expression.

Initializing bean and scope

  • To initialize the bean by passing parameters during their instantiation a body is provided for the <jsp:useBean> tag.
  • <jsp:useBean id="address" scope="session" class="AddressBean" >
  • <% address.setStreet("123 Main St. "); %>
  • </jsp:useBean>
  • If the bean is already present in the given scope then the body of useBean tag is not executed. 
  • It is executed only when the JSP engine creates a new instance of the bean class.
  • When a <jsp:useBean> declaration is enclosed inside a Java programming language block using scriptlets and a pair of curly braces, the scope of the variable declared by the action also gets restricted to the enclosing block

<jsp:setProperty> Attributes

  • The name attribute
  • The name attribute identifies a particular instance of an existing bean and is mandatory. 
  • The bean must be declared already by <jsp:useBean> action, and the value of the name attribute must be the same as the value of the id attribute of <jsp:useBean>.
  • The property attribute
  • The property attribute specifies the property of the bean to be set and is a mandatory attribute. 
  • The JSP engine calls the setXXX() method on the bean based on the specified property.
  • The value attribute
  • The value attribute specifies the new value to be set for the bean’s property.
  • The attribute can also accept a request-time attribute expression.

<jsp:setProperty>: param Attribute

  • The param attribute specifies the name of request parameter.  
  • If the request contains the specified parameter, then the value of that parameter is used to set the bean’s property.
  • The value and param attributes are never used together and are both optional.
  • If neither of the two is specified, it is equivalent to having the same value for both param and property, and the JSP engine searches for a request parameter with the name that is same as the property attribute.
  • If the parameter is not present in the request, or has an empty value, the <jsp:setProperty> action has no effect and the property retains its original value.
  • Setting all the properties of the bean in one action using request parameters is as follows:
  • <jsp:setProperty name="address" property="*" /> 
  • If the request parameter has multiple values, then only the first value is used.

Accessing using <jsp:getProperty>

  • The <jsp:getProperty> action is used to retrieve and print the values of the bean properties to the output stream.
  • Both name and property attributes, are mandatory. 
  • The name attribute specifies the name of the bean instance as declared by a previous <jsp:useBean> action and the property attribute specifies the property whose value is to be printed.
  • When the JavaBean has non-string properties <jsp:setProperty> and <jsp:getProperty> perform the necessary type conversions automatically even in case of the request parameters for <jsp:setProperty>.

Using indexed properties

  • Indexed properties can be used to associate multiple values to a single property.
  • Setting index properties automatically from request parameter.
  • <jsp:setProperty name=“class" property=“students" />
  • The jsp engine creates an array of datatype in javabean with the length of the array equal to the number of request parameter values for the parameter named “students”.
  • Setting index properties using a request-time expression and explicitly pass an array of the desired type.
  • <%!  char myStudents[] = {‘James', ‘Ryan', ‘Steve' };   %>
  • <jsp:setProperty name=“class“ property=“students“          
  •       value="<%=myStudents%>" />
  • All the parameter values are converted to their respective types for each individual element of the array for setting the index properties.
  • When we get indexed properties from a bean using the <jsp:getProperty> action, it is equivalent to calling out.print(property-type []), where propertytype is the data type of the indexed property.

Custom Tags

  • Custom tags help to move the presentation logic outside the JSP pages into independent Java classes, thus centralizing the implementation of the presentation logic and increasing maintainability.
  • They are similar to the standard JSP actions and follow the same XML syntax format.
  • Custom tags enable to avoid duplicating the presentation logic; removing the scriptlets making the pages less cluttered and easier to read.
  • In Custom Tags the onus of thread safety is on the container which maintains a pool of instances of Tag Handler, and reuses the appropriate instance from a pool with its attributes reset.

Terms Used In Custom Tags

  • Tag handler: It is a runtime, container-managed object that evaluates custom actions during execution of JSP page.
  •  A tag handler is a Java class that implements one of the tag interfaces—Tag, IterationTag, or BodyTag.
  • Tag library: It is a collection of actions that encapsulate some functionality to be used from within a JSP page.
  • A set of custom tags that work together and help to solve a recurring requirement is called a tag library.
  • Tag library descriptor: The meta-information of custom tags such as their tag handler classes, their tag library location and their usage, is stored in a file called the tag library descriptor (TLD).
  •  

Informing Custom Tag Libraries

  • A new tag library is imported into a JSP page using a taglib directive:
  • <%@ taglib prefix="test" uri="sampleLib.tld" %>
  • The above declaration inform the engine that the page uses custom tags with the prefix test and that these tags are described in the file sampleLib.tld.
  • The JSP container maintains a map between the URIs used in taglib directives and the actual physical location of the TLD files in the file system.
  • The JSP engine reads the URI in taglib directive and refers to its internal map to find the corresponding TLD file location.
  • The actual TLD file resides in the WEB-INF directory or in META-INF directory in the JAR file.
  • The JSP container will recognize either a directory or a JAR, as a path to a TLD file called the TLD resource path. 

Explicit Mapping of TLD

  • The container reads the user-defined mapping entries present in the deployment descriptor called explicit mapping.
  • The <taglib> element of deployment descriptor file, associates user-defined URIs with TLD resource paths.
  • Each <taglib> element associates one URI with one location. It has elements as follows:
  • <taglib-uri>: It is the user-defined URI.
  •  <taglib-location>: It is the TLD resource path.
  • Sample <taglib> element in deployment descriptor.
  • <taglib>
  • <taglib-uri> http://www.abc.com/learn </taglib-uri>
  • <taglib-location> /myLibs/studyKit.tld </taglib-location>
  • </taglib>

Resolving URIs to TLD file locations

  • The JSP pages can refer to the mapped URIs using the taglib directives:
  • <%@ taglib prefix="study“ uri="http://www.abc.com/" %>
  • The JSP engine checks its taglib map to see if a mapping exists for the uri attribute of the taglib directive and uses the value of  corresponding <taglib-location> to locate the actual TLD file.
  • If the <taglib-location> value is a root-relative URI then the location is assumed to be relative to the web application’s document root directory.
  • If the <taglib-location> value is a non-root relative URI, the JSP engine prepends /WEB-INF/ to the URI and assumes location to be relative to the web application’s document root directory.

Understanding Custom Tag Prefix

  • Every standard JSP action tag has a tag name that is made up of two parts, a prefix and an action.
  • The general syntax of Custom tags is as follows:
  • <myPrefix:myCustomAction>
  • The prefix differentiates between tags that belong to different libraries.
  • In order to avoid conflicts between user defined tags and standard tags there are certain restrictions on value of prefix attribute such as custom tag libraries cannot use prefix values such as jsp, jspx, java, javax, servlet, sun, and sunw.

Using Custom Tags

  • Following are different types of custom tags in a JSP page:
  • • Empty tags
  • • Tags with attributes
  • • Tags with JSP code
  • • Tags with nested tags

Empty tags

  • Empty tags do not have any body content.
  • They are written in two ways. 
  • 1) They can consist of a pair of opening and closing tags without anything in between:
  • <prefix:tagName></prefix:tagName>
  • 2) They can also be formatted as a single self-tag:
  • <prefix:tagName />
  • The self-tag has a forward slash / at the end of the tag.

Tags with attributes

  • The Tags accept the attribute value to perform its operation.
  • Similar to std tags, custom tags also can have attributes defined as mandatory. 
  • The JSP engine flags an error at translation time when the mandatory attributes are not specified.
  • In case of unspecified non-mandatory attributes, the tag handler uses the default values depend on its implementation.
  • The attribute values can be either constants or JSP expressions.
  • <prefix:tagName  attrib1=“fixedValue” attrib2=“<%=someJSPExpresson%>”  attrib3= ... />
  • The expressions are evaluated at request time and passed to the corresponding tag handler.
  •  

Tags with JSP code and nested tags

  • A tag may contain JSP code enclosed within the opening and closing tags called the body content of the tag. Body content can be any valid JSP code.
  • Non-empty tags can also include other custom tags in their body and are called nested tags.
  • The opening and closing tags of a nested tag and its enclosing tag cannot overlap.
  • <test:switch conditionValue=
  • '<%=request.getParameter("action")%>' >
  • <test:case caseValue="sayHello"> Hello! </test:case>
  • <test:default> I am Dumb!!! </test:default>
  • </test:switch> 
  •

  • The <taglib> element is the top-level, or root, element of the tag library descriptor.

  • There must be at least one <tag> subelement and zero or more <listener> elements.

  • The JSP engine does not attempt to execute the body and passes it as is to the tag handler at request time

Life Cycle of Custom Tag

  • The setPageContext() method is the first method called by the JSP container passing the pageContext implicit object of the JSP page.
  • The JSP container calls the setParent() method on the child tag and passes it a reference to the parent tag.
  •  The getParent() method is called by a child tag and not directly by the JSP container to get the reference to the parent tag.
  • If a custom tag has any attributes, then for each attribute, the JSP engine calls the appropriate setter method to set its value at request time.
  • After calling setter methods, the container calls the doStartTag() method on the tag, giving the tag handler a chance to do initial computations and to verify whether or not the attribute values passed in the setter methods are valid. 
  • If the initialization fails, the method may throw a JspException or a subclass of JspException, such as JspTagException

Life Cycle of Custom Tag

  • The doStartTag() method decides whether or not to continue evaluating its body content by returning the constant EVAL_BODY_INCLUDE or SKIP_BODY in order to execute the body content or skip the body content.
  • After the body of the tag is evaluated or skipped the container calls the doEndTag() method.
  • The doEndTag() gives the tag handler a chance to do the final cleanup for the tag and then decides whether or not to continue evaluating the rest of the JSP page by returning either EVAL_PAGE or SKIP_PAGE.
  • The container finally calls the release() method on the handler class when the tag handler object is no longer required and is called only once regardless of every call to doEndTag().
  • A single instance of the tag handler is used to handle all of the occurrences of custom tags.
